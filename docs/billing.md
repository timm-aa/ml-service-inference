# Логика биллинга и отказоустойчивость

## Баланс и транзакции

- Кредиты хранятся в `users.balance_credits`.
- Каждое изменение баланса отражается записью в `credit_transactions` с итоговым `balance_after`.
- Базовая стоимость одного предсказания задаётся ключом `prediction_base_cost_credits` в таблице `billing_config` (по умолчанию 10). Её можно менять без пересборки приложения.

## Скидка по уровню лояльности (вариант Б)

- У пользователя есть `loyalty_tier_id`, уровень задаётся в `loyalty_tiers` с полями `min_predictions_monthly` и `discount_percent`.
- Фактическая цена (как в `discounted_cost` в `app/services/pricing.py`): при `base_credits ≤ 0` — `0`; иначе скидка в процентах ограничивается \(d = \min(100,\max(0,\mathrm{discount\_percent}))\), затем \(\max\bigl(1,\ \mathrm{round}(\mathrm{base}\cdot(100-d)/100)\bigr)\) кредитов.
- Уровень пересчитывается фоновой задачей Celery Beat **1-го числа каждого месяца** по числу **успешных** предсказаний за **предыдущий календарный месяц** (`worker.tasks.recalculate_loyalty_monthly`).

## Когда списываются кредиты

1. Клиент создаёт задачу предсказания (`POST /predictions`). API проверяет, что оценочная стоимость (с учётом текущей скидки) не превышает баланс.
2. Задача выполняется в Celery **асинхронно**, HTTP не блокируется ожиданием модели.
3. После **успешного** `predict` в одной транзакции БД выполняется списание и создаётся транзакция типа `prediction_charge` с ссылкой `reference_type=prediction_job`, `reference_id=<uuid задачи>`.
4. Уникальный ключ `(reference_type, reference_id)` гарантирует **идемпотентность**: повторное списание за ту же задачу невозможно.

## Ошибки и сбои

- Если inference падает, статус задачи — `failed`, списания нет.
- Если между проверкой баланса в API и списанием в воркере баланс упал, воркер получит `Insufficient credits at billing time`, задача станет `failed`, списание не произойдёт (или сработает идемпотентная защита при гонках).
- Рекомендуется мониторить очередь Celery и метрики Prometheus (`/metrics`), чтобы видеть рост `failed`.

## Пополнение (заглушка платёжного шлюза)

- `POST /billing/payment-callback` принимает сумму, внешний идентификатор платежа и HMAC-подпись `SHA256-HMAC(secret, external_id:amount_credits)`.
- Секрет задаётся переменной окружения `PAYMENT_WEBHOOK_SECRET`. Дубликат `external_payment_id` отклоняется уникальным ограничением на транзакции.
